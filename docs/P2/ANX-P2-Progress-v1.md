A new `RuntimeError::StringIndexOutOfBounds` variant (same shape as the array one, distinct message) covers `charAt`/`substring` bounds violations.
- **Codegen â€” the real work, per the plan's own framing:**

  - Both `Type::Str` and `Type::Array(_)` are now struct-valued in LLVM, which meant `codegen_print`'s dispatch (previously keyed on the LLVM value's shape â€” `PointerValue` meant "string") had to switch to keying on sema's resolved type instead, since the two are no longer distinguishable by shape alone. Same fix needed in the ternary `phi` codegen (merged the `Str`/`Array` match arms â€” both do the identical `into_struct_value()` dance now).
  - `charAt`/`substring` bounds-checking is inline in codegen (`emit_runtime_guard` + a new `anx_panic_str_oob`), mirroring the array index check almost line-for-line per the plan's explicit instruction â€” the C runtime shim (`anx_str_concat`, `anx_str_char_at`, `anx_str_substring`, `anx_str_equals`) does the actual malloc+memcpy work only once codegen has already confirmed the indices are valid.
